# FarmApp ŌĆö Firebase Deployment Runbook

This runbook walks a developer with the `firebase` CLI installed through deploying
the Firestore rules, Storage rules, and (eventually) indexes for FarmApp, plus
building the Flutter app for device. It is the single source of truth for the
one-time setup per Firebase project and per-deploy steps.

> **Hard rule** ŌĆö never deploy rules without first verifying you are on the
> intended project. Run `firebase projects:list` and `firebase use --add`
> *before* any `firebase deploy ŌĆ”` invocation. Deploying rules to the wrong
> project can lock real users out or open production data. Always check.

The Firebase project ID currently wired into the app (from `firebase.json`) is
**`farm-app-mvp`**. Update this runbook if/when a staging or production project
is added.

## 1. Prerequisites

### 1.1. Local tooling

- **Node.js Ōēź 18** (required by `firebase-tools`).
- **Firebase CLI**:
  ```bash
  npm install -g firebase-tools
  firebase --version   # expect 13.x or newer
  ```
- **FlutterFire CLI** (only needed if you regenerate `firebase_options.dart`):
  ```bash
  dart pub global activate flutterfire_cli
  ```
- **Flutter SDK** matching `pubspec.yaml`'s constraint, with `flutter doctor`
  clean for the platforms you intend to build.

### 1.2. Authentication

```bash
firebase login           # interactive browser login
firebase login:list      # confirm the right Google account is active
```

If you have multiple accounts, `firebase login:use <email>` switches the
active account. Use the account that has at least **Editor** on the target
Firebase project (or `Firebase Rules Admin` + `Cloud Datastore Index Admin`
at a minimum).

### 1.3. Target Firebase project

A Firebase project must already exist. From the FarmApp repo root:

```bash
firebase projects:list                # see what your account can access
firebase use --add                    # interactively bind a project alias
firebase use                          # print the currently-active project
```

The current default is `farm-app-mvp` (see `firebase.json` ŌåÆ
`flutter.platforms.android.default.projectId`). There is no `.firebaserc`
checked in, so the active project is whatever your local CLI is set to.

> Repeat: confirm the active project before *any* deploy command. The CLI does
> not prompt ŌĆö `firebase deploy` will silently use whichever project is active.

### 1.4. Firebase products enabled on the project

In the Firebase Console (`https://console.firebase.google.com/project/<project-id>/`)
verify each of the following is enabled. These are one-time clicks per project.

- **Authentication**
  - Sign-in method ŌåÆ **Email/Password** ŌåÆ enabled.
  - (Email link / Google / etc. are not used by v1.)
- **Cloud Firestore**
  - Mode: **Native** (not Datastore).
  - Location: a region close to the Philippines ŌĆö `asia-southeast1` (Jakarta)
    or `asia-east1` (Taiwan) are reasonable picks. Pick once; cannot be
    changed later.
- **Cloud Storage**
  - The default bucket (`<project-id>.appspot.com`) is created automatically
    when you first open Storage in the Console. Confirm it exists.

### 1.5. Flutter app wiring

The Flutter app reads its Firebase configuration from
`lib/firebase_options.dart`, which is generated by `flutterfire configure`.

If you switch to a new Firebase project, regenerate it:

```bash
flutterfire configure \
  --project=<project-id> \
  --platforms=android,ios \
  --out=lib/firebase_options.dart
```

This rewrites `lib/firebase_options.dart`, `android/app/google-services.json`,
and the iOS `GoogleService-Info.plist`. Commit the changes.

## 2. First-time deploy

Do these once per Firebase project. Each command below assumes you have
already run `firebase use <project-id>` and confirmed the output of
`firebase use` shows the intended project.

### 2.1. Verify the active project

```bash
firebase use
firebase projects:list                    # cross-reference
```

If the project shown is wrong:

```bash
firebase use --add                        # pick the right one
```

### 2.2. Deploy Firestore rules

```bash
firebase deploy --only firestore:rules
```

Source file: `firestore.rules` (path configured in `firebase.json`).

Expected output ends with `+  Deploy complete!`. Open the Console ŌåÆ
Firestore ŌåÆ Rules to confirm the timestamp matches your deploy.

### 2.3. Deploy Storage rules

```bash
firebase deploy --only storage
```

> **Note:** use `storage`, **not** `storage:rules`. The CLI parses
> `storage:<name>` as a deploy target (multi-bucket setup); a single-bucket
> project has none configured, so `storage:rules` errors with
> `Could not find rules for the following storage targets: rules`.
> `firestore:rules` *is* valid because firestore has named sub-flags
> (`rules`, `indexes`); storage doesn't.

Source file: `storage.rules`. The v1 rule limits uploads to **5 MB per file**
under `/farms/{farmId}/...` for authenticated farm members only.

Confirm in Console ŌåÆ Storage ŌåÆ Rules.

### 2.4. Deploy Firestore indexes

The repo currently has **no `firestore.indexes.json`** ŌĆö `firebase.json` only
declares `firestore.rules`. The app relies on Firestore's automatic
single-field index creation plus a small number of collection-group queries
that Firestore can serve without composite indexes.

**Collection-group queries the app issues:**

| Query | File | Notes |
|---|---|---|
| `collectionGroup('pens').where('farmId', ==)` | `lib/src/features/areas/data/area_repository.dart` | Single-field auto-index on `farmId`. |
| `collectionGroup('line_items').where('pigId', ==)` | `lib/src/features/sales/data/sale_repository.dart` | Single-field auto-index on `pigId`. |
| `collectionGroup('health_records')` | `lib/src/features/profitability/application/profitability_providers.dart` | No `where` ŌĆö no index needed. |
| `collectionGroup('invitations').where('email', ==)` | `lib/src/features/team/data/team_repository.dart` | Single-field auto-index on `email`. |
| `collectionGroup('members').where(FieldPath.documentId, ==)` | `lib/src/features/team/data/team_repository.dart` | Single-field auto-index on `__name__`. |
| `collectionGroup('mortality_record')` | `lib/src/features/pigs/data/mortality_repository.dart` | No `where` ŌĆö no index needed. |
| `collectionGroup('farrowing_records')` | `lib/src/features/pigs/data/farrowing_repository.dart` | No `where` ŌĆö no index needed. |
| `collectionGroup('maintenance_records')` | (planned) | No `where` expected. |

**Default behaviour: nothing to deploy.** Firestore auto-creates single-field
collection-group indexes the first time a query runs that requires one, and it
auto-exempts the rest. If a query fails at runtime, Firestore returns an error
that includes a one-click URL to the Console where you can create the missing
index in 30 seconds.

> **If you ever add a composite index** (multi-field `.where(...).orderBy(...)`),
> add it to a new `firestore.indexes.json` in the repo root, add
> `"indexes": "firestore.indexes.json"` under `firestore` in `firebase.json`,
> and deploy with `firebase deploy --only firestore:indexes`.

### 2.5. Smoke-verify

After the rules deploys finish:

1. Run the app on a device (see ┬¦5).
2. Sign up a fresh user from the in-app flow.
3. Create a farm (this writes `farms/{farmId}` + `farms/{farmId}/members/{uid}`).
4. Try to read another (non-member) farm's docs from a test harness ŌĆö should fail.
5. Upload a photo from the app ŌĆö should succeed and land under
   `farms/{farmId}/...` in Storage.

If any of these fail, jump to ┬¦6.

## 3. Per-deploy steps (rules changes)

This is the path you take whenever you edit `firestore.rules` or
`storage.rules`.

1. Edit the rules file.
2. (Optional) Local syntax check using the CLI's parser. Older `firebase-tools`
   versions do **not** support `--dry-run` for rules; run the deploy itself
   against a staging project, or skip and rely on the syntax check the deploy
   does automatically before publishing.
   ```bash
   # If your CLI supports it:
   firebase deploy --only firestore:rules --dry-run
   # If not, syntax is validated as part of the real deploy.
   ```
3. Confirm the active project ŌĆö yes, again:
   ```bash
   firebase use
   ```
4. Deploy:
   ```bash
   firebase deploy --only firestore:rules
   # or
   firebase deploy --only storage      # NOT storage:rules ŌĆö see ┬¦2.3
   ```
5. Smoke-test the affected role flow:
   - Auth/profile changes ŌåÆ sign in/out, edit profile photo.
   - Pig/farrowing/health changes ŌåÆ log a record as Worker.
   - Sales/expenses/purchases ŌåÆ as Owner/Manager, create a sale.
   - Team/members/invitations ŌåÆ invite, accept, remove.
6. If a rule unintentionally blocks a flow, **revert the rules file**, deploy
   the previous version, then debug locally. Do not leave the project on a
   broken ruleset.

## 4. Wiping test data

For pre-launch testing only. **Never** run these against a production project.

### 4.1. Firestore data

Firebase Console ŌåÆ Firestore ŌåÆ Data tab. For a clean slate:

- Delete the **`farms`** collection (right-click ŌåÆ Delete collection).
- Delete the **`users`** collection.

The Console handles recursive deletion of all sub-collections (members,
invitations, pigs, breeding_records, farrowing_records, health_records,
mortality_record, batches, areas/pens, equipment/maintenance_records, tasks,
shifts, activity, supplies, supply_movements, purchases/line_items, expenses,
sales/line_items).

> Console deletion of large collections runs in the background and can take
> minutes. The UI shows progress.

### 4.2. Auth users

Firebase Console ŌåÆ Authentication ŌåÆ Users tab. Either:

- Leave users intact (so test accounts can keep signing in), or
- Delete users individually with the trash icon next to each row.

There is no bulk-delete in the Console UI; for full wipes use the Admin SDK
or `gcloud firestore` from a workstation that has appropriate credentials.

### 4.3. Storage objects

Firebase Console ŌåÆ Storage ŌåÆ Files. Right-click the `farms/` prefix ŌåÆ Delete.

## 6. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `PERMISSION_DENIED: Missing or insufficient permissions` on Firestore | Rules not deployed, or signed-in user doesn't have a `farms/{farmId}/members/{uid}` document. | Re-run `firebase deploy --only firestore:rules`. In Console ŌåÆ Firestore, verify `farms/{farmId}/members/{uid}` exists with `removedAt == null`. |
| Photos won't upload (`storage/unauthorized`) | Storage rules not deployed, or user is not a farm member, or file > 5 MB. | Deploy with `firebase deploy --only storage` (not `storage:rules` ŌĆö see ┬¦2.3). Check file size on the client before upload (PhotoService should already enforce this). Verify membership doc as above. |
| Collection-group query throws `FAILED_PRECONDITION` with a URL in the message | A single-field collection-group index is missing. | Open the URL printed in the device console ŌĆö Firestore takes you straight to a "Create index" page. Click create; the index builds in 1-5 minutes. The next query then succeeds. |
| `App is not authorized to use Firebase Authentication` | `firebase_options.dart` (or `google-services.json` / `GoogleService-Info.plist`) is from a different project than the one Auth is enabled on. | Rerun `flutterfire configure --project=<project-id>`. Commit and rebuild. |
| `[firebase_auth/network-request-failed]` | Device offline, or DNS blocked. | Try a different network. Auth requires online; offline sign-in is not supported by Firebase. |
| Invitation accept fails with permission denied | The accept flow expects a self-create on `members/{uid}` with `_inviteId` matching an existing `invitations/{id}` doc. | Check the invitation doc still exists and matches the accepting user's email. |
| Worker can't toggle equipment status | The rule only permits workers to change `status` + `updatedAt`. The repository must not also touch other fields in the same write. | Inspect the WriteBatch; remove unintended field updates. |
| `Cloud Firestore API has not been used in project ... before or it is disabled.` | Firestore wasn't enabled in the Console for this project. | Open Console ŌåÆ Firestore ŌåÆ Create database. Choose Native mode. |
| `gsutil`/`gcloud` works but Console upload fails | You're authenticated as one identity in the CLI and another in the browser. | Match accounts, or use the browser's account picker. |

For deeper inspection, `firebase emulators:start --only firestore,storage,auth`
runs a local emulator suite you can point the app at with
`FirebaseFirestore.instance.useFirestoreEmulator('localhost', 8080)` and
friends. This isn't wired up by default; add it if you need offline testing
without touching the real project.

## 7. Rules contract checklist

A one-page cheat sheet of who can do what. Canonical source is
**Spec ┬¦8** in `docs/superpowers/specs/2026-05-14-swine-crm-foundation-design.md`
plus the spec B ┬¦8 delta. This table is for verification only ŌĆö when it
disagrees with the spec, the spec wins.

Roles: **O** = Owner ┬Ę **M** = Manager ┬Ę **W** = Worker ┬Ę **V** = Vet.
Ō£ō = full read+write where applicable. R = read only. ŌĆö = no access.

| Resource | O | M | W | V | Notes |
|---|---|---|---|---|---|
| `users/{uid}` (self) | Ō£ō | Ō£ō | Ō£ō | Ō£ō | Self-only; no cross-user reads. |
| `farms/{farmId}` | Ō£ō | R | R | R | Only Owner can update farm doc. |
| `farms/{farmId}/members` | Ō£ō | Ō£ō | R | R | O/M manage team; W/V read-only. Self-create allowed when accepting an invitation. |
| `farms/{farmId}/invitations` | Ō£ō | Ō£ō | R | R | Invitee can read their own pending invite by email; can update status ŌåÆ `accepted`. |
| `farms/{farmId}/areas` + `pens` | Ō£ō | Ō£ō | R | R | O/M only for writes. |
| `farms/{farmId}/equipment` | Ō£ō | Ō£ō | status-toggle | R | Workers may write only `status` + `updatedAt`. |
| `equipment/.../maintenance_records` | Ō£ō | Ō£ō | create | R | Workers can create; only O/M update or delete. |
| `farms/{farmId}/pigs` | Ō£ō | Ō£ō | Ō£ō | R | O/M/W can edit; only O/M can delete. |
| `pigs/.../breeding_records` | Ō£ō | Ō£ō | create + edit-own-within-24h | R | |
| `pigs/.../farrowing_records` | Ō£ō | Ō£ō | create + edit-own-within-24h | R | |
| `pigs/.../health_records` | Ō£ō | Ō£ō | create + edit-own-within-24h | Ō£ō create + edit-own-within-24h | Vet can write health only. |
| `pigs/.../mortality_record` | Ō£ō | Ō£ō | create + edit-own-within-24h | R | |
| `farms/{farmId}/batches` | Ō£ō | Ō£ō | Ō£ō | R | |
| `farms/{farmId}/tasks` | Ō£ō | Ō£ō | create autoGenerated + complete-only | R | Any member may flip `status`, `completedBy`, `completedAt`. |
| `farms/{farmId}/shifts` | Ō£ō | Ō£ō | R | R | |
| `farms/{farmId}/activity` | append-only | append-only | append-only | append-only | Anyone can append their own entry; nobody can update or delete. |
| `farms/{farmId}/supplies` | Ō£ō | Ō£ō | currentStock-only | R | Workers limited to `currentStock` + `updatedAt`. |
| `farms/{farmId}/supply_movements` | Ō£ō | Ō£ō | create consumption only | R | Immutable ledger; no updates or deletes. |
| `farms/{farmId}/purchases` (+ `line_items`) | Ō£ō | Ō£ō | ŌĆö | ŌĆö | Financial; O/M only. |
| `farms/{farmId}/expenses` | Ō£ō | Ō£ō | ŌĆö | ŌĆö | Financial; O/M only. |
| `farms/{farmId}/sales` (+ `line_items`) | Ō£ō | Ō£ō | ŌĆö | ŌĆö | Financial; O/M only. |
| Storage `/farms/{farmId}/**` | Ō£ō | Ō£ō | Ō£ō | Ō£ō | Any farm member; 5 MB max per file. |

**Verification recipe** ŌĆö for each role:

1. Sign in as a user with that role on a known farm.
2. From the app, attempt one action per row above. Successful actions should
   succeed silently; denied actions should surface the correct denial UX
   (toast, disabled button, or hidden affordance ŌĆö never a stack trace).
3. From Firestore Rules Playground (Console ŌåÆ Firestore ŌåÆ Rules ŌåÆ Playground),
   replay the highest-risk denials with a synthetic request to make sure the
   rule, not the UI, is doing the gating.
